# 阶段三：规模化阶段实施计划（18 个月+）

## 阶段目标

当团队扩张到 10 人以上、用户量超过 5 万时，单体架构的部署周期、扩缩容灵活性、数据隔离需求开始成为瓶颈。此阶段的核心目标是：**通过绞杀者模式渐进式拆分微服务，实现独立部署、独立扩缩容、独立团队治理**。

---

## 一、拆分信号识别

进入规模化阶段的触发条件（满足 3 个以上）：

| 信号 | 说明 |
|------|------|
| 部署周期 > 30 分钟 | 单体代码量过大，构建时间过长 |
| 数据库成为瓶颈 | 单库 QPS > 5000，连接数不足 |
| 扩缩容冲突 | AI 推理需要 GPU，用户服务不需要，但部署绑定 |
| 团队规模 > 10 人 | 多团队并行开发，代码合并冲突频繁 |
| 故障影响面大 | 一个模块的 bug 导致全站不可用 |
| 技术栈分歧 | 不同团队需要不同的技术选型 |

## 二、微服务拆分策略

### 2.1 拆分顺序优先级

```
优先级 1：AI 推理服务
├── 原因：GPU 资源需求与 Web 服务完全独立
├── 收益：独立扩缩容，GPU 成本大幅降低
└── 风险：中等（内部服务，接口稳定）

优先级 2：计费/积分服务
├── 原因：财务合规要求，数据需要独立审计
├── 收益：隔离敏感数据，简化合规审计
└── 风险：低（核心模块，稳定，接口明确）

优先级 3：用户/会员服务
├── 原因：用户数据是多个服务的依赖
├── 收益：统一用户视图，简化权限管理
└── 风险：中等（跨服务调用多）

优先级 4：通知服务
├── 原因：非核心功能，拆分成本低
├── 收益：解耦通知逻辑，减少主服务负载
└── 风险：低（异步通知，容错要求低）

优先级 5：场景/工具配置服务
├── 原因：配置频繁变更，不应触发全站部署
├── 收益：独立迭代，快速上线
└── 风险：低（配置服务，接口简单）
```

### 2.2 绞杀者模式（Strangler Fig Pattern）

通过 API 网关逐步将流量从单体导向新服务，而非大爆炸式重写：

```
阶段一：部署新服务，与单体并行运行
         ↓
阶段二：通过网关将 10% 流量切换到新服务
         ↓
阶段三：监控稳定性，逐步提升到 50%、100%
         ↓
阶段四：单体中对应模块下线
```

### 2.3 拆分后的服务架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                          API 网关层                                  │
│                    (Kong / 自建 Gateway)                            │
│         认证 │ 鉴权 │ 限流 │ 路由 │ 监控                           │
└─────────────────────────────────────────────────────────────────────┘
          │           │           │           │           │
          ↓           ↓           ↓           ↓           ↓
    ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ 用户服务 │ │ 计费服务 │ │ AI 推理  │ │ 通知服务 │ │ 配置服务 │
    │          │ │          │ │  服务    │ │          │ │          │
    │ - 注册   │ │ - 积分   │ │          │ │ - 邮件  │ │ - 场景  │
    │ - 登录   │ │ - 会员   │ │ - 模型   │ │ - 短信  │ │ - 工具  │
    │ - 权限   │ │ - 账单   │ │ - 路由   │ │ - 推送  │ │ - 工作流│
```

## 三、AI 推理服务拆分

### 3.1 服务职责

AI 推理服务是**最先拆分、最先收益**的服务，因为它：

- 需要 GPU 资源，与 Web 服务扩缩容策略完全不同
- 是成本最高的模块，独立部署便于成本核算
- 内部调用为主，接口相对稳定

### 3.3 与计费服务集成

AI 推理服务通过 **gRPC** 或 **HTTP** 调用计费服务：

```python
# ai-inference-service/core/billing_client.py
import httpx

class BillingClient:
    def __init__(self, billing_service_url: str):
        self.url = billing_service_url
    
    async def deduct_points(
        self, 
        user_id: str, 
        amount: int, 
        task_id: str
    ) -> bool:
        """调用计费服务扣减积分"""
        async with httpx.AsyncClient() as client:
            try:
                response = await client.post(
                    f"{self.url}/internal/deduct",
                    json={
                        "user_id": user_id,
                        "amount": amount,
                        "reference_id": task_id,
                        "reference_type": "ai_task"
                    },
                    timeout=5.0
                )
                return response.status_code == 200
            except httpx.TimeoutException:
                # 降级：任务标记为待审核，人工处理
                return False
    
    async def refund_points(self, user_id: str, amount: int, task_id: str):
        """任务失败时退还积分"""
        async with httpx.AsyncClient() as client:
            await client.post(
                f"{self.url}/internal/refund",
                json={
                    "user_id": user_id,
                    "amount": amount,
                    "reference_id": task_id
                }
            )
```

## 四、计费/积分服务拆分

### 4.1 服务职责

计费服务是**最敏感、最需要独立审计**的服务：

- 涉及资金变动，必须保证强一致性
- 财务合规要求数据独立存储和审计
- 其他服务（AI、通知等）都需要调用扣费/退款接口

### 4.2 核心接口

## 六、服务间通信

### 6.1 同步通信（HTTP/gRPC）

适用于需要实时响应的场景：

```python
# 用户服务调用计费服务获取会员信息
class UserService:
    def __init__(self, billing_client: BillingClient):
        self.billing = billing_client
    
    async def get_user_with_membership(self, user_id: str):
        user = await self.user_repo.get(user_id)
        membership = await self.billing.get_membership(user_id)
        
        return {
            **user,
            "membership": membership
        }
```

### 6.2 异步通信（事件总线）

适用于解耦和最终一致性的场景：

```python
# 用户服务发布事件
class UserService:
    def __init__(self, event_bus: EventBus):
        self.event_bus = event_bus
    
    async def on_user_created(self, user: User):
        # 创建用户后，发布事件
        await self.event_bus.publish(
            "user.created",
            {
                "user_id": user.id,
                "email": user.email,
                "created_at": user.created_at.isoformat()
            }
        )

# 计费服务订阅事件
class BillingEventHandler:
    async def handle_user_created(self, event: dict):
        # 为新用户创建积分账户
        await self.points_service.create_account(
            user_id=event["user_id"]
        )
        # 发放注册奖励
        await self.points_service.earn(
            user_id=event["user_id"],
            amount=100,  # 注册赠送
            source="SIGNUP_BONUS"
        )
```

### 6.3 事件总线选型

| 方案 | 适用场景 | 成本 |
|------|---------|------|
| Redis Streams | 小规模，服务 < 10 | 低 |
| RabbitMQ | 中等规模，需要复杂路由 | 中 |
| Kafka | 大规模，高吞吐 | 高 |
| AWS SNS/SQS | AWS 生态 | 按量 |

### 7.2 跨服务数据查询

避免跨库 JOIN，设计时考虑：

```python
# 方案一：聚合 API（推荐）
# 用户中心需要展示用户信息 + 积分余额
@router.get("/api/v1/users/{user_id}/dashboard")

        "user": user,

# 方案二：数据冗余（最终一致性）
# 在用户服务冗余会员等级信息（异步同步）
@event_handler("membership.changed")
```
